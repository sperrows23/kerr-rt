# Summary: GRay — Massively Parallel GPU-Based Geodesic Integrator (arXiv 1303.5057v1)

## (1) Identification

**Authors:** Chi-Kwan Chan, Dimitrios Psaltis, Feryal Özel (University of Arizona / NORDITA / Harvard-Smithsonian CfA)
**Year:** 2013 (draft dated November 27, 2024 in the preprint header; original arXiv submission 2013)
**Title:** "GRay: A Massively Parallel GPU-Based Code for Ray Tracing in Relativistic Spacetimes"
**Venue:** The Astrophysical Journal (submitted); arXiv:1303.5057v1

---

## (2) Core Contribution

GRay is a CUDA C/C++ geodesic integrator that traces photon null geodesics in arbitrary curved spacetimes on NVIDIA GPUs using the stream-processing (SIMD) paradigm. It achieves >300 GFLOP peak in single precision, reaching ~1 ns per photon per timestep, and is 50–100× faster than the best equivalent CPU code (Ray) and >1000× faster than the semi-analytical code Geokerr. The code is open-source (GPL v3). The key application demonstrated is a systematic parameter study of Kerr black hole shadow/photon-ring properties across 16 spin values × 19 inclinations, integrating ~3×10⁸ geodesics.

## (3) Key Equations & Quantities

**Coordinate system:** Boyer-Lindquist (BL) coordinates (t, r, θ, φ) for the Kerr metric. Units: geometric (G = M = c = 1).

**Geodesic equations integrated (second-order form, eqs. 9–12 in Psaltis & Johannsen 2012):**
The paper integrates the full second-order geodesic equations (not the first-order Carter-constant reduction), making the algorithm applicable to non-Kerr spacetimes lacking Petrov-type D.

**Horizon radius** (event horizon termination boundary):
$$r_\mathrm{bh} \equiv 1 + \sqrt{1 - a^2}$$
where *a* is the dimensionless spin parameter. Integration stops at r = r_bh + δ (δ ~ 10⁻⁶).

**Adaptive step size** (step-size formula, eq. 2 in the paper — rendered as `formula-not-decoded` in the source):
$$\Delta\lambda \propto \frac{\Delta}{r - r_\mathrm{bh}}$$
with Δ ~ 1/32 a user-provided parameter; ensures the step shrinks as the photon approaches the horizon.

**Null-geodesic normalization monitor** (pole-treatment trigger):
$$\xi \equiv g_{\mu\nu} k^\mu k^\nu$$
where k^μ is the photon 4-momentum. This should equal 0 for null geodesics; the paper monitors |ξ + 1| > ε (ε ~ 10⁻³) for a timelike normalization proxy to detect integration errors near the poles. (The paper uses ξ for the full 4-velocity normalization diagnostic — the exact sign convention in the source references timelike normalization = −1.)

**θ-remapping** (domain enforcement):
$$\theta \mapsto \theta \bmod \pi, \quad \theta \in [0, \pi]$$
Enforces the polar coordinate to stay in its valid domain after each step.

**Image-plane coordinates:** (α₀, β₀) — 2D Cartesian on the observer image plane at r = 1000 M. Local polar coordinate (R, ϑ) defined by:
$$\alpha_0 = R\cos\vartheta, \quad \beta_0 = R\sin\vartheta$$

**Photon ring observables** (all DIRECTLY USABLE):
- **Horizontal displacement:** D = (α₀_max + α₀_min) / 2 — offset of ring center from spacetime geometric center
- **Average radius:** ⟨R⟩ — mean radius of the photon ring on the image plane, where R ≡ [(α₀ − D)² + β₀²]^(1/2)
- **Asymmetry parameter:** A — quantifies non-circularity of the ring

**Empirical fitting formulae** for ⟨R⟩ and A as functions of spin a and inclination i (degrees) are provided in §4 but the exact closed-form coefficients are in `formula-not-decoded` blocks and cannot be quoted exactly from this source.

**CUDA thread/block grid formula:**
$$n_\mathrm{grid} = \left\lfloor \frac{n + n_\mathrm{block} - 1}{n_\mathrm{block}} \right\rfloor$$
with n_block = 64 (fixed); ensures n_grid × n_block ≥ n total photons.

## (4) Methods / Algorithms

**Integrator:** Standard 4th-order Runge-Kutta (RK4), integrating the second-order geodesic equations (following Psaltis & Johannsen 2012). Not the Carter-reduced first-order system — this is a deliberate design choice enabling non-Kerr generality.

**Parallelization scheme:** One CUDA thread per geodesic. Photon states stored as array-of-structures; an in-block data transpose via shared memory converts this to structure-of-arrays layout for optimal GPU memory bandwidth. Block size fixed at n_block = 64 (over-subscribed relative to physical stream processors per multiprocessor to hide memory latency). Grid size computed with the standard idiomatic CUDA ceiling-division formula above.

**Coordinate handling:**
- BL coordinates; adaptive step size proportional to (r − r_bh)⁻¹ near the horizon, controlled by user parameter Δ ~ 1/32.
- Photon integration halted at r = r_bh + δ (δ ~ 10⁻⁶) to avoid crossing the horizon.
- θ remapped modulo π to enforce [0, π] domain after each step.

**Pole singularity treatment (critical numerical-stability trick):**
The standard RK4 fails when two substeps land on opposite sides of θ = 0 or π — truncation errors do not cancel and instead amplify. Three levels tested:
1. *Naive RK4* (blue): photon incorrectly deflected to infinity.
2. *Euler fallback* (green): fall back to 1st-order forward Euler whenever a geodesic crosses the pole; fixes the failure but introduces larger truncation error and can fail when a full step lands near the pole.
3. *Production scheme* (red, DIRECTLY USABLE): monitor |ξ + 1| > ε ~ 10⁻³; if triggered, re-integrate the offending step with 1st-order Euler at reduced step size Δλ′/9. The factor 1/9 is chosen so that (i) the single low-order step does not significantly increase absolute error, and (ii) the Euler step does not itself land on the pole. Subsequent steps are shifted past the pole. Validated against 3×10⁸ geodesics with zero pole failures when Δ = 10⁻⁶.

**Real-time visualization:** CUDA-OpenGL interoperability; frame buffer on GPU mapped to CUDA so no host-device transfer. OpenGL Shading Language (GLSL) handles coordinate transform and sprite rendering.

**Performance:** On NVIDIA Tesla M2090, single precision: 1.15 ns/step/photon; double precision: 7.15 ns/step/photon. On GeForce GT 650M, single precision: 3.27 ns/step/photon. CPU baseline: Ray (C, RK4) = 357–693 ns/step/photon; Geokerr (FORTRAN, semi-analytic) = 23,000–43,800 ns/step/photon. GRay is ~50× faster than Ray and >1000× faster than Geokerr per integration step.

## (5) Results & Key Parameters

- **Spin study:** 16 spin values with (1 − a) evenly spaced in log scale; a = 0 to a = 0.999.
- **Inclination study:** 19 values, i = 0°, 5°, 10°, …, 90°.
- **Image plane:** r = 1000 M; polar grid 6000 × 181 rays in (R, ϑ) ∈ (1.5, 7.5) × [0, π].
- **Total geodesics:** 16 × 19 × 6000 × 181 ≈ 3×10⁸.
- **Photon ring average radius ⟨R⟩:** weakly dependent on spin; significant asymmetry only for a ≳ 0.99 and large inclination.
- **Key finding:** The near-circular shape of the photon ring at high spin results from cancellation of Kerr ellipsoidal geometry by frame-dragging, exact to quadrupole order (Johannsen & Psaltis 2010).
- **Kernel launch overhead:** ~20 ms (MacBook Pro), up to ~0.5 s (some Linux configurations) — independent of photon count; relevant only for small batch sizes.
- **1024×1024 image:** computable in a few seconds on Tesla M2090.

## (6) Relevance to a Kerr + Accretion-Disk Offline Renderer

**High relevance.** Several elements are directly applicable:

1. **RK4 geodesic integrator pattern:** The adaptive-step RK4 with horizon cutoff (r_bh + δ) and θ-remapping is a battle-tested template for the project's Taichi geodesic kernel. The step-size formula (∝ 1/(r − r_bh)) and δ ~ 10⁻⁶ stopping criterion can be adopted directly.

2. **Pole singularity fix (DIRECTLY USABLE):** The production "re-integrate at Δλ/9 with Euler on |ξ+1| > ε" trick is a concrete, validated solution to a known failure mode when photons pass near the spin axis — essential for a near-extremal a = 0.999 renderer where polar geodesics are physically relevant.

3. **CUDA parallelization pattern:** One thread per geodesic; fixed block size 64; in-block transpose for memory coalescing. Directly maps to Taichi's CUDA backend thread model.

4. **Normalization monitor ξ:** Using g_μν k^μ k^ν as an in-flight accuracy diagnostic (|ξ + 1| > ε triggers corrective action) is a robust quality guard for the project's integrator.

5. **Coordinate note:** GRay uses Boyer-Lindquist coordinates throughout, while the project's active path is Cartesian Kerr-Schild (CKS). The algorithms are transferable but the metric components and Christoffel symbols differ — the pole singularity in BL (θ = 0, π) does not appear in CKS, making the pole-treatment section less critical but still informative about geodesic robustness strategies.

6. **Photon ring geometry:** The fitting formulae for ⟨R⟩ and A(a, i) at a = 0.999 provide ground-truth reference values for validating the renderer's shadow and photon-ring output.

7. **Performance baseline:** The 1–7 ns/step/photon on 2012-era NVIDIA hardware is a useful reference floor; modern GPUs (RTX 5060 / sm_120) should substantially exceed this.
