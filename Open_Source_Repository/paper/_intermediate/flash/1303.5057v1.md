# Implementation Report: Massive Parallel GPU-Based Ray Tracing in Relativistic Spacetimes

## 1. Identification
* **Title:** GRay: A Massively Parallel GPU-Based Code for Ray Tracing in Relativistic Spacetimes
* **Authors:** Chi-Kwan Chan, Dimitrios Psaltis, Feryal Özel
* **Year:** 2013 (arXiv preprint arXiv:1303.5057v1, compiled draft version November 2024)
* **Venue:** Published in *The Astrophysical Journal*, 777:13 (2013)

## 2. Core Contribution
The paper introduces **GRay**, a massively parallel GPU-based geodesic integrator designed to trace billions of photon trajectories in curved spacetimes. Its core contributions include:
1. **Direct Geodesic Integration:** Integrating second-order geodesic equations directly rather than relying on first-order integrals of motion. This design enables the integrator to remain extensible to non-Kerr metrics.
2. **Robust Coordinate Pole Handling:** A numerical fallback algorithm designed to handle the coordinate singularity at the rotation poles ($\theta = 0, \pi$) of Boyer-Lindquist coordinates, preventing artificial numerical reflections of light rays.
3. **CUDA-OpenGL Interoperability:** A zero-copy visualization pipeline where geodesic data resides entirely in GPU VRAM, allowing real-time rendering via custom OpenGL Shading Language (GLSL) shaders.
4. **Semi-Analytic Fits for Kerr Shadows:** Highly accurate empirical fitting formulae for the average radius, displacement, and asymmetry of Kerr black hole shadows as a function of dimensionless spin $a$ and observer inclination $i$.

## 3. Key Equations & Quantities
All equations are presented in geometric units ($G = M = c = 1$). The spacetime coordinates are Boyer-Lindquist coordinates $(t, r, \theta, \phi)$.

### Coordinate Singularities and Integration Steps
* **Event Horizon Radius ($r_{\text{bh}}$):**
  $$r_{\text{bh}} = 1 + \sqrt{1 - a^2}$$
  *For a near-extremal black hole with $a = 0.999$, $r_{\text{bh}} \approx 1.0447$. This is directly usable to define the absorption boundary in the renderer.*

* **Adaptive Affine Step Size ($\Delta\lambda$):**
  $$\Delta\lambda = \Delta \frac{r - r_{\text{bh}}}{r}$$
  Where $\Delta \sim 1/32$ is a user-supplied accuracy parameter. 
  *Directly usable in the ray integration loop to scale step sizes down near the horizon and scale up at large distances.*

* **Termination Boundary:**
  $$\text{Stop if } r \le r_{\text{bh}} + \delta$$
  Where $\delta \sim 10^{-6}$ prevents the integrator from crossing the horizon and causing coordinate divergence.

* **Polar Boundary Reflection:**
  To keep $\theta$ strictly in the physical domain $[0, \pi]$, the standard reflection mapping is applied:
  $$\text{If } \theta < 0: \quad \theta \to -\theta, \quad \phi \to \phi + \pi$$
  $$\text{If } \theta > \pi: \quad \theta \to 2\pi - \theta, \quad \phi \to \phi + \pi$$
  *Directly usable for maintaining coordinate validity when rays cross the rotation axis.*

* **Constraint Violation Monitor ($\xi$):**
  $$\text{Monitor: } | \xi + 1 | > \varepsilon$$
  Where $\xi \equiv p^\mu p_\mu$ represents the normalized geodesic constraint (configured to remain constant at $-1$ in the paper's integration parametrization), and $\varepsilon \sim 10^{-3}$.
  *Directly usable as a trigger for numerical adaptive refinement.*

### Image Plane Coordinate Transformations
The image plane is placed at a distance $r_0 = 1000$ from the black hole. The Cartesian coordinates $(\alpha_0, \beta_0)$ on the observer's screen are related to the local shadow polar coordinates $(R, \vartheta)$ by:
$$\alpha_0 = D + R \cos\vartheta$$
$$\beta_0 = R \sin\vartheta$$
Where $D$ is the horizontal displacement of the shadow's center from the coordinate origin, and $\vartheta$ is the polar angle on the image plane.

### Analytical Shadow Fitting Formulae (Directly Usable for Verification)
The average radius of the photon ring $\langle R \rangle$ and the asymmetry parameter $A$ are modeled using the following relations (where the observer inclination angle $i$ must be supplied in **degrees**):

$$\langle R(a, i) \rangle = R_0(a) + R_1(a) i^2 + R_2(a) i^4$$
With:
$$R_0(a) = 3\sqrt{3} - 0.203 a^2 + 0.045 a^4$$
$$R_1(a) = (-0.186 a^2 + 0.086 a^4) \times 10^{-4}$$
$$R_2(a) = (-0.313 a^2 + 0.055 a^4) \times 10^{-8}$$

The asymmetry parameter is given by:
$$A(a, i) = A_1(a) i^3 + A_2(a) i^5$$
With:
$$A_1(a) = (0.288 a^2 + 0.089 a^3) \times 10^{-5}$$
$$A_2(a) = (0.165 a^2 - 0.141 a^3) \times 10^{-9}$$

## 4. Methods/Algorithms

### Parallel Execution & Memory Management
* **Thread Mapping:** Each geodesic is independent and maps to a single CUDA thread.
* **Block Size Selection:** Thread block size $n_{\text{block}}$ is fixed to $64$. This over-subscribes the physical GPU stream processors, enabling zero-overhead warp context-switching to hide global memory latency.
* **Grid Dimensioning:** The number of blocks in the grid is dynamically calculated via:
  $$n_{\text{grid}} = \left\lfloor \frac{n + n_{\text{block}} - 1}{n_{\text{block}}} \right\rfloor$$
  Where $n$ is the total number of traced photons.
* **Coalesced Memory Access:** Photon states are initially laid out as an Array-of-Structures (AoS) for ease of logical manipulation. An in-block data transposition is executed using fast on-chip shared memory to convert this layout into a Structure-of-Arrays (SoA) to guarantee coalesced global memory transactions.

### Integration Loop and Robust Pole Fallback
Geodesics are advanced using a standard 4th-order Runge-Kutta (RK4) integration scheme. When a ray passes close to the rotation axes ($\theta = 0$ or $\pi$), standard RK4 fails because its intermediate substeps are evaluated on opposite sides of the polar coordinate singularity. This invalidates the local smoothness assumption, preventing the cancellation of low-order truncation errors and resulting in the ray artificially bouncing back to infinity.

The paper resolves this coordinate issue with the following algorithm:
1. Complete an RK4 step and calculate the constraint value $\xi$.
2. Check if the error exceeds the threshold: $| \xi + 1 | > \varepsilon$.
3. If the threshold is exceeded:
   * Discard the inaccurate RK4 step.
   * Roll back the ray state to the previous step.
   * Take a 1st-order forward Euler step with a drastically smaller step size:
     $$\Delta\lambda' = \frac{\Delta\lambda}{9}$$
   * Resume standard RK4 integration on subsequent steps. This shifts the subsequent full evaluation points away from the exact pole while preserving overall numerical stability.

## 5. Results & Key Parameters
* **Performance Figures:** On a single NVIDIA Tesla M2090 GPU, GRay achieves an integration performance of **1.15 ns per photon per time step** in single precision, and **7.15 ns per photon per time step** in double precision.
* **Comparative Speed:**
  * GRay is **$\sim 50\times$ faster** per step than a highly optimized serial CPU RK4 C-code (`Ray`, $356.67 \text{ ns/step/photon}$).
  * GRay is **$\sim 20,000\times$ faster** per step than serial semi-analytical CPU codes (`Geokerr`, $23,000 \text{ ns/step/photon}$).
* **Kerr Shadow Characteristics:** For spins up to $a \approx 0.99$, the shadow remains highly circular across all observer inclinations. Significant asymmetric distortion (the classic "flattened" D-shape) is only manifested for extreme spins ($a \ge 0.999$) coupled with high observer inclinations ($i \ge 60^\circ$).

## 6. Relevance to a Kerr + Accretion-Disk Offline Renderer

### 1. High-Performance Ray Casting Design
For offline rendering, massive parallelization is key. The team should implement the geodesic solver using the exact thread mapping proposed by the paper ($1 \text{ ray} = 1 \text{ thread}$, $n_{\text{block}} = 64$). 

### 2. Eliminating Database Bottlenecks
The paper proves that storing precomputed geodesics on disk to reconstruct rays for rendering is highly inefficient due to storage latency. For a coarse $100^2$ grid with 100 points per ray, a database spanning various spins and inclinations requires $\sim 11\text{ GB}$ of storage, limiting ray-processing performance to $\sim 10^3\text{ rays/sec}$ due to disk read times. Integrating geodesics on-the-fly using the GPU is vastly superior, enabling more than $10^6\text{ rays/sec}$.

### 3. Near-Extremal Spin ($a = 0.999$) Horizon Handling
With $a=0.999$, the event horizon is extremely close to the coordinate singularity at $r=1$ ($r_{\text{bh}} \approx 1.0447$). The adaptive step size $\Delta\lambda = \frac{1}{32}\frac{r-r_{\text{bh}}}{r}$ is highly recommended to prevent step-overs or numerical crashes as rays graze the horizon or orbit within the photon ring before escaping or falling in.

### 4. Over-the-Pole Disk Rendering
When rendering accretion disks at high inclinations (e.g., $i = 85^\circ$), a significant portion of the back of the disk is lensed over the black hole's pole towards the observer. Without the paper’s robust pole fallback algorithm (discarding failed RK4 steps and falling back to a forward Euler step with $\Delta\lambda/9$), the renderer will exhibit severe visual artifacts (black pixels or noise) near the rotation axis of the black hole.

### 5. Verification of Geodesic Solvers
The analytic fits for $\langle R(a, i) \rangle$ and $A(a, i)$ provide a robust unit test for the renderer. For a simulated camera at $r = 1000$, $a = 0.999$, and $i = 60^\circ$, the rendered shadow boundary must match the fitted average radius of $\approx 4.88$ and the predicted asymmetry $A \approx 0.045$.
